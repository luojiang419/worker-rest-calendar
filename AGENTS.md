# Codex 仓库执行规则

## 必须先读

按顺序阅读：

1. `README.md`
2. `docs/01_PRODUCT_REQUIREMENTS.md`
3. `docs/03_SCHEDULE_ENGINE.md`
4. `docs/05_TECH_ARCHITECTURE.md`
5. `docs/07_UI_UX_SPEC.md`
6. 当前阶段对应的 `codex/PHASE_XX_*.md`

## 不可改变的领域定义

- 大周 = 周六、周日休息。
- 小周 = 周六上班、周日休息。
- 周一是排班周起点。
- 手动单日覆盖优先级最高。
- 法定节假日数据覆盖基础班制，但低于手动覆盖。
- 加班是附加记录，不替换“工作/休息”的基础含义。
- “调休上班”和“调休休息”必须是两个可区分状态。

## 开发纪律

- 每次只完成一个 Phase，不跨阶段大改。
- 排班引擎先写纯 Dart 测试，再接 UI。
- UI 组件必须从设计令牌读取颜色、圆角、阴影、间距和动画时长。
- 浅色与暗黑模式必须同步实现，不允许先写死浅色颜色。
- 平台特有能力必须封装在 adapter/service 中，避免散落条件判断。
- 每个 Phase 完成后运行：格式化、静态检查、单元测试、已有集成测试。
- 未通过测试时不得勾选 `PROGRESS.md`。
- 不把密钥、服务端 URL、证书或用户数据提交到仓库。

## 每阶段输出格式

Codex 完成阶段时必须报告：

1. 修改文件清单。
2. 已实现行为。
3. 执行过的命令和结果。
4. 尚存风险或平台限制。
5. 下一阶段前置条件。
